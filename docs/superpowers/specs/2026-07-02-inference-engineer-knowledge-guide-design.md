# 推理工程师面试知识体系 Guide 设计

## 目标

基于五个大模型推理、训练基础设施、多模态算子和 Agent 基础设施岗位（来自六张截图），提炼一套面向个人面试准备的通用知识体系。

最终文档不是岗位描述汇编，也不是按天安排的学习计划。它应帮助读者：

- 理解推理工程师岗位解决的核心问题；
- 建立从计算机基础到生产推理系统的完整知识结构；
- 区分通用能力与不同岗位方向的专项能力；
- 判断每项知识在面试中需要掌握到什么深度；
- 从新 JD 的关键词识别岗位侧重点。

## 交付形式

- 最终文档：`docs/inference-engineer-knowledge-guide.md`
- 格式：Markdown
- 语言：中文，技术名词保留通用英文写法
- 读者：准备大模型推理工程师相关面试的个人

## 内容结构

### 1. 岗位本质

从性能、成本和稳定性三类目标解释推理工程师的职责，并说明算法、硬件和系统之间需要联合优化。

### 2. 六层核心知识体系

知识体系按依赖关系自底向上组织：

1. 计算机与系统基础；
2. 大模型与生成模型原理；
3. 加速硬件与高性能算子；
4. 模型级推理优化；
5. 推理引擎与运行时；
6. 分布式推理与生产系统。

每一层使用统一模板：

- **目标**：该层解决的问题；
- **核心知识**：应建立联系的关键概念；
- **典型技术**：岗位中常见的框架、工具和实现；
- **关键权衡**：性能设计中的主要矛盾；
- **面试深度**：候选人应达到的理解与实践层级；
- **边界说明**：通用能力和专项能力的分界。

### 3. 岗位分支

在共同知识底座之上，单独描述三类专项方向：

- 多模态与生成模型推理；
- 大规模训练推理基础设施；
- Agent 与业务 AI 基础设施。

分支只补充差异化能力，不重复核心知识体系。

### 4. 能力深度标准

使用统一的五级标准描述掌握程度：

1. 了解概念和用途；
2. 能解释核心原理与关键权衡；
3. 能实现最小可用版本；
4. 能分析性能并完成针对性优化；
5. 能设计、部署和负责生产级系统。

### 5. JD 信号解读

将岗位关键词映射到主要岗位类型：

- CUDA、Triton、TileLang、FlashAttention 等对应算子与硬件方向；
- vLLM、SGLang、TensorRT-LLM 等对应推理引擎方向；
- 调度、通信、高并发、弹性、可用性等对应分布式系统方向；
- 量化、蒸馏、稀疏化、剪枝等对应模型压缩方向；
- Diffusion、视频、图像、语音等对应多模态方向；
- Agent、评测、特征工程、数据闭环等对应业务 AI 基础设施方向。

### 6. 面试知识检查表

以问题形式检查知识覆盖和概念连接。检查表不包含学习周期、课程推荐或按日任务。

### 7. JD 特征归纳

用一张表总结五个岗位共同强调的能力、出现频率、对应知识层和典型技术。该表用于说明抽象结论的来源，不逐份复述原始 JD。

## 写作原则

- 以稳定概念组织内容，工具名称作为例子，不让文档依赖某个框架版本；
- 解释知识之间的因果和依赖，避免关键词堆砌；
- 明确区分基础要求、高频要求和方向性加分项；
- 不包含公司、联系人、发布日期等招聘元信息；
- 不虚构岗位描述未支持的硬性学历、年限或技术要求；
- 保持可扩展性，后续新增 JD 时优先映射到现有层级，只有出现新的能力边界时才新增分类。

## 完成标准

- 五个岗位中反复出现的技术主题均能映射到文档中的明确位置；
- 六层知识体系边界清晰，内容没有明显重复；
- 三类岗位分支能解释不同 JD 的差异；
- 每层均包含目标、核心知识、典型技术、关键权衡、面试深度和边界说明；
- 读者能够据此判断自己的知识盲区，并识别新岗位的主要侧重点。

## 非目标

- 不制作逐日或逐周学习计划；
- 不提供题库答案或完整课程；
- 不评价具体公司或岗位优劣；
- 不根据这些岗位样本推断招聘成功率。
