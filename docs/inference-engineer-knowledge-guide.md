# 推理工程师面试知识体系

这份 Guide 从多份大模型推理、训练基础设施、多模态算子与 Agent 基础设施岗位描述中提取共同能力，目标不是覆盖某一家公司的特定技术栈，而是建立一套可以迁移到不同岗位的知识地图。

## 如何使用这份 Guide

先用六层核心体系检查共同基础，再根据目标 JD 选择专项分支。准备每个主题时，不应停留在“听说过某个名词”，而应依次回答四个问题：

1. 它解决什么瓶颈？
2. 它为什么有效，代价是什么？
3. 它在完整推理链路中位于哪里？
4. 如何用指标、实验或 profiling 证明优化有效？

真正有区分度的能力不是背出工具列表，而是能够沿着“模型结构 → 算子 → 运行时 → 分布式服务 → 业务指标”定位问题，并在精度、延迟、吞吐、成本和稳定性之间做出可解释的取舍。

## 一、岗位本质：在约束下交付更高效的模型能力

推理工程师处在算法、硬件和系统的交界处。工作的核心，是让模型在真实流量与真实硬件约束下稳定运行，并持续改善以下指标：

- **延迟**：首 Token 延迟（TTFT）、逐 Token 延迟（TPOT）、端到端响应时间及尾延迟；
- **吞吐**：单位时间内处理的请求数、Token 数或多模态样本数；
- **成本**：每请求、每 Token 或每任务消耗的 GPU 时间、显存和集群资源；
- **精度与质量**：量化、稀疏化或近似计算不能让模型质量跌破业务阈值；
- **资源利用率**：计算单元、显存、带宽、网络和多机集群是否被有效使用；
- **稳定性**：高并发、突发流量、硬件故障和跨地域部署下是否仍然可用；
- **研发效率**：新模型、新算子和新硬件能否快速接入、验证和上线。

这些指标经常彼此冲突。例如，扩大 batch 可以提高吞吐，却可能增加排队延迟；低比特量化可以降低显存与计算成本，却可能损伤精度；更激进的请求合批可以提高 GPU 利用率，却会增加调度复杂度和尾延迟。因此，推理优化不是追求单点 benchmark，而是围绕目标场景建立约束、测量瓶颈、实施优化、验证收益并防止回归。

典型工作闭环可以抽象为：

> 明确 SLO 与成本目标 → 建立基线 → 分解端到端耗时 → 定位瓶颈 → 选择优化层级 → 实施与验证 → 生产监控与持续迭代

## 二、六层核心知识体系

六层知识自底向上相互依赖。底层决定“机器实际上如何执行”，中层决定“模型如何被高效运行”，上层决定“优化能否在生产环境中转化为稳定收益”。

### 1. 计算机与系统基础

**目标**

理解推理系统运行所依赖的软件与系统机制，能够分析 CPU、内存、网络、并发和分布式组件对端到端性能的影响。

**核心知识**

- C/C++ 的内存模型、对象生命周期、并发原语和性能成本；
- Python 的运行时开销、扩展机制以及与 C++/CUDA 的边界；
- 数据结构与算法复杂度，尤其是队列、缓存、哈希、堆和图；
- Linux 进程、线程、虚拟内存、文件系统、I/O 与性能观测；
- 计算机网络、RPC、连接管理、序列化、拥塞与超时；
- 高并发系统中的排队、背压、限流、重试和负载均衡；
- 分布式系统中的一致性、故障模型、可用性和可扩展性。

**典型技术**

Linux、C/C++、Python、Go/Rust/Java、gdb、perf、eBPF、RPC、共享内存、异步 I/O、容器与集群调度系统。

**关键权衡**

- 开发效率与运行效率；
- 并发度与锁竞争；
- 复制带来的可用性与一致性成本；
- 重试带来的成功率与流量放大；
- 通用抽象与关键路径上的极致性能。

**面试深度**

至少应能解释线程与进程、同步与异步、零拷贝、缓存局部性、RPC 超时和背压。更高阶的回答应能把这些机制连接到推理服务，例如解释 Python 调度层为何不一定是瓶颈、何时需要 C++ 热路径，以及请求队列如何影响 P99 延迟。

**边界说明**

这是所有推理岗位的共同底座。无需把自己准备成通用后端专家，但必须能够分析推理系统涉及的并发、网络、内存与故障问题。

### 2. 大模型与生成模型原理

**目标**

理解模型的计算结构和生成过程，从而判断时间、显存和通信消耗来自哪里，并知道哪些优化会影响模型质量。

**核心知识**

- Transformer 的 attention、MLP、归一化、残差连接和位置编码；
- 自回归生成中的 prefill、decode、KV Cache 与采样；
- 参数量、激活量、序列长度、batch size 与计算量/显存量的关系；
- Multi-Head、Multi-Query、Grouped-Query Attention 的差异；
- Mixture of Experts（MoE）的路由、专家并行和负载均衡；
- Diffusion/DiT 的迭代去噪过程、采样步数和条件控制；
- 图像、视频、语音、3D 等多模态模型中的编码器、解码器与跨模态模块；
- 训练、微调与推理在计算图、状态和并行策略上的不同需求。

**典型技术**

PyTorch、Transformer、KV Cache、RoPE、GQA/MQA、MoE、Diffusion、DiT、VAE、Speculative Decoding。

**关键权衡**

- 上下文长度与显存/计算成本；
- 模型容量与部署成本；
- 采样质量与生成速度；
- 稠密模型的规则计算与 MoE 的通信、负载不均；
- 多模态质量与输入尺寸、帧数、采样步数。

**面试深度**

应能手算或估算关键模块的计算量和显存占用，解释 prefill 与 decode 的性能特征为何不同，并说明 KV Cache 为什么会成为长上下文和高并发场景的核心约束。面向多模态岗位，还应能指出 Diffusion 与 LLM 推理链路的主要差异。

**边界说明**

推理岗位通常不要求从零训练最先进模型，但要求理解模型结构到足以预测性能、选择优化方案并判断质量风险。

### 3. 加速硬件与高性能算子

**目标**

理解算子如何映射到加速硬件，能够判断瓶颈是计算、显存带宽、访存模式、同步还是通信，并据此优化关键 Kernel。

**核心知识**

- GPU/NPU 的执行模型、计算单元、线程层级和调度机制；
- 寄存器、共享内存、缓存与全局显存的层次结构；
- 内存合并访问、数据复用、bank conflict、occupancy 与指令吞吐；
- Roofline 模型、算术强度以及 compute-bound/memory-bound 判断；
- MatMul、Grouped MatMul、Conv2D、归一化、激活和 attention 等核心算子；
- FlashAttention 的分块、在线 softmax 与 I/O 优化思想；
- 算子融合、布局变换、流水化和异步拷贝；
- 单算子 profiling、数值正确性验证和性能回归分析；
- CUDA 之外的异构算力及其算子生态。

**典型技术**

CUDA、cuBLAS、cuDNN、NCCL、Triton、TileLang、AscendC、HIP、Bang、Cutlass/Cute DSL、Nsight Systems、Nsight Compute。

**关键权衡**

- 通用性与硬件特化；
- occupancy 与单线程资源使用；
- 算子融合收益与编译/维护复杂度；
- 更低精度与数值稳定性；
- 峰值吞吐与不同 shape 下的稳定表现。

**面试深度**

基础层面应能解释 GPU 线程模型、显存层次和 coalesced access。进阶层面应能根据 profiling 指标定位瓶颈，说明一个 Triton/CUDA Kernel 的分块策略，并分析 FlashAttention、矩阵乘或 MoE Grouped MatMul 的优化路径。优秀候选人还应能讨论数值误差、shape 泛化和多硬件适配。

**边界说明**

算子岗位需要达到“能实现并优化”的深度；引擎或系统岗位至少要达到“能读懂 profiling、识别算子瓶颈并与算子团队协作”的深度。

### 4. 模型级推理优化

**目标**

在不重写整个服务系统的前提下，改变模型表示、计算图或执行精度，降低模型的计算、显存和通信成本。

**核心知识**

- PTQ/QAT、权重量化、激活量化、KV Cache 量化和混合精度；
- INT8、INT4、FP8、BF16/FP16 等数值格式及校准方法；
- 结构化/非结构化稀疏、剪枝和稀疏算子支持；
- 知识蒸馏、模型裁剪与小模型替代；
- 图优化：常量折叠、死代码消除、算子融合和布局优化；
- 编译优化：图捕获、shape specialization、代码生成和自动调优；
- 投机解码、前缀缓存等减少有效计算的算法；
- 优化后的精度评估、回归检测和场景化验收。

**典型技术**

GPTQ、AWQ、SmoothQuant、FP8、TensorRT、torch.compile、XLA/TVM 类编译思想、稀疏化、蒸馏、剪枝、Speculative Decoding。

**关键权衡**

- 精度与压缩率；
- 校准成本与模型泛化；
- 静态 shape 优化与动态请求适配；
- 理论 FLOPs 降低与硬件实际加速；
- 优化收益与模型版本迭代成本。

**面试深度**

应能解释量化误差从何而来、为什么权重低比特不必然带来等比例加速、校准集如何影响结果，以及图优化和 Kernel 优化的边界。进阶回答应包含质量指标、硬件支持、端到端 benchmark 和回退方案。

**边界说明**

模型级优化连接算法与系统。它不是只看压缩率的离线实验，最终必须由真实硬件上的延迟、吞吐、显存和质量共同验收。

### 5. 推理引擎与运行时

**目标**

把模型、优化算子和硬件组织成高效运行时，在动态请求下管理显存、批处理、调度、并行和执行图。

**核心知识**

- 模型加载、权重布局、设备映射与执行计划；
- continuous batching、动态 batch 和迭代级调度；
- KV Cache 的分配、分页、复用、淘汰与前缀缓存；
- prefill/decode 分离与 chunked prefill；
- 请求优先级、抢占、公平性和尾延迟控制；
- Tensor Parallel、Pipeline Parallel、Data Parallel、Expert Parallel 和 Sequence Parallel；
- CUDA Graph、异步执行、流水线与计算通信重叠；
- 多卡通信拓扑和 collective operation；
- 扩展新模型、新算子、新硬件和新量化格式的接口设计。

**典型技术**

vLLM、SGLang、TensorRT-LLM、Megatron-LM、PagedAttention、NCCL、CUDA Graph、ZeRO、TP/PP/DP/EP/SP。

**关键权衡**

- 吞吐与单请求延迟；
- batch 效率与调度公平性；
- KV Cache 利用率与碎片/管理开销；
- 并行度与通信成本；
- 框架通用性与模型/硬件特化；
- 激进调度与可预测的 P99。

**面试深度**

应能画出一个 LLM 推理引擎的数据流，解释请求如何进入调度器、如何形成 batch、如何分配 KV Cache、如何在多卡上执行。进阶层面应能比较 vLLM、SGLang、TensorRT-LLM 的设计取向，并针对长上下文、高并发或多 LoRA 等场景提出调度与内存方案。

**边界说明**

推理引擎关注单机或一组设备内“如何高效执行模型”；跨实例流量治理、弹性和高可用主要属于下一层。

### 6. 分布式推理与生产系统

**目标**

把推理引擎扩展为可承载真实业务流量的服务，在多机、多集群和多地域环境中满足 SLO，并控制资源成本。

**核心知识**

- 网关、路由、模型服务、调度器、Worker 和存储等系统组件；
- 请求队列、负载均衡、批处理边界和流式返回；
- 多实例/多机并行、模型副本和异构资源调度；
- 容量规划、弹性扩缩容、GPU 超卖和资源配额；
- 故障检测、熔断、降级、重试、容灾与发布回滚；
- 指标、日志、trace、profiling 与在线性能归因；
- 多租户隔离、优先级、配额和成本核算；
- 离线评测、线上实验、质量监控和性能回归；
- 跨地域部署与网络、数据、容量约束。

**典型技术**

容器与集群调度、服务发现、RPC、分布式队列、Tracing、Metrics、Autoscaling、Canary、模型网关、容量与成本平台。

**关键权衡**

- 高利用率与突发流量余量；
- 高可用与冗余成本；
- 全局调度收益与跨地域网络开销；
- 多租户共享与性能隔离；
- 快速发布与回归风险；
- 平均吞吐与尾延迟/SLO。

**面试深度**

应能设计一个端到端在线推理服务，明确组件边界、核心指标、故障处理和扩容策略。更高阶回答应能结合排队论、GPU 利用率、模型并行和业务优先级讨论容量与成本，并说明如何从线上指标回溯到引擎或算子瓶颈。

**边界说明**

这一层强调生产责任。高性能 benchmark 只有转化为稳定的线上吞吐、延迟和成本收益，才算完成推理优化闭环。

## 三、岗位分支

所有岗位共享六层底座，但会在不同层投入不同深度。以下分支只列差异化能力。

### 1. 多模态与生成模型推理

重点通常位于“模型原理—算子—运行时”三层：

- 理解 Diffusion/DiT、VAE、视觉编码器、音频模块和跨模态 attention；
- 分析图像分辨率、视频帧数、采样步数对计算量与显存的影响；
- 优化 Conv2D、Attention、MatMul、Grouped MatMul 等密集算子；
- 处理动态 shape、长流水线、多阶段模型和中间激活峰值；
- 完成 GPU/NPU 等异构卡适配，以及算子覆盖、精度对齐和性能迁移；
- 将量化、稀疏、蒸馏和算子融合应用到图像、视频、语音或 3D 生成。

这类岗位的面试通常比纯 LLM 服务岗位更关注模型计算图差异、专用算子、显存峰值以及端到端 pipeline 优化。

### 2. 大规模训练推理基础设施

重点通常位于“硬件—引擎—分布式系统”三层：

- 千卡级 GPU 集群、拓扑感知调度和多机通信；
- TP、PP、DP、EP、SP、ZeRO 等并行与切分方法；
- 计算与通信重叠、通信热点、straggler 和负载均衡；
- 编译优化、子图匹配、图执行与新硬件后端接入；
- 资源池化、弹性调度、GPU 超卖、任务编排和利用率治理；
- 高并发、高可靠、高扩展系统的架构设计与故障恢复；
- 训练和推理框架之间可复用的算子、通信与调度基础设施。

这类岗位往往偏好分布式系统、并行计算、编译器、网络或存储背景，因为规模扩大后，单卡算子性能只是系统效率的一部分。

### 3. Agent 与业务 AI 基础设施

重点通常位于“模型服务—评测—数据—业务系统”的连接处：

- Agent CLI、Skill、工具调用、知识库和工作流等通用组件；
- 特征/样本的数据采集、清洗、标注、消费、版本与质量监控；
- 离线与在线评测、统一榜单、回归测试、负例分析和效果归因；
- 模型/Agent 迭代中的数据闭环与可追溯性；
- vLLM 等推理框架在业务环境中的适配、性能优化和成本治理；
- 高并发、高可用服务与线上问题排查；
- 安全、灵活性、质量和工程效率之间的平衡。

这类岗位未必要求极深的 CUDA Kernel 开发，但要求更完整的工程闭环：模型不仅要“跑得快”，还要可评估、可追踪、可迭代并对业务结果负责。

## 四、能力深度标准

面对 JD 中的“熟悉”“掌握”“有经验”等模糊措辞，可以用以下五级标准评估自己：

| 级别 | 能力定义 | 面试中的典型表现 |
| --- | --- | --- |
| 1 | 了解概念和用途 | 能给出准确的一句话定义，知道它解决什么问题 |
| 2 | 能解释核心原理与关键权衡 | 能画结构、讲数据流，并说明收益与代价 |
| 3 | 能实现最小可用版本 | 能写出简化实现，说明正确性验证方法 |
| 4 | 能分析性能并完成针对性优化 | 能读 profiling，提出假设，用 benchmark 验证优化 |
| 5 | 能设计、部署和负责生产级系统 | 能处理规模、故障、SLO、成本、发布与持续演进 |

岗位中的基础项通常至少需要达到 2 级；简历中明确写过的项目应达到 3 至 4 级；声称主导过生产系统，则需要能够以 5 级深度回答。

## 五、如何从 JD 判断岗位侧重点

职位名称都可能写成“推理优化工程师”，真正的工作重心要从高频动词、性能对象和加分项判断。

| JD 信号 | 主要方向 | 应重点准备 |
| --- | --- | --- |
| CUDA、Triton、TileLang、FlashAttention、MatMul | 算子与硬件 | GPU 架构、访存、分块、融合、profiling、数值正确性 |
| vLLM、SGLang、TensorRT-LLM、PagedAttention | 推理引擎 | batching、调度、KV Cache、执行图、框架扩展 |
| 调度、通信、高并发、弹性、高可用、多地域 | 分布式系统 | 并行策略、RPC、排队、容灾、容量、SLO |
| 量化、蒸馏、稀疏化、剪枝、编译优化 | 模型级优化 | 精度损失、硬件支持、图优化、端到端收益 |
| Diffusion、视频、图像、语音、3D | 多模态推理 | 模型 pipeline、专用算子、动态 shape、显存峰值 |
| Agent、评测、特征工程、数据闭环、知识库 | 业务 AI 基础设施 | 数据质量、评测回归、工具链、服务化、成本治理 |
| 千卡集群、训练框架、ZeRO、TP/PP/SP/EP | 训练推理 Infra | 通信、拓扑、并行、任务调度、集群利用率 |
| 新硬件接入、GPU/NPU/TPU/异构卡 | 异构适配 | 算子覆盖、编译后端、精度对齐、性能迁移 |

还可以通过岗位动词判断深度：

- “使用、接入、部署”偏应用与工程集成；
- “优化、分析、调优”要求性能分析与实验闭环；
- “研发、构建、二次开发”要求实现框架或关键组件；
- “主导、架构、建设”通常要求跨层设计和生产责任。

## 六、面试知识检查表

如果以下问题无法在白板上清晰回答，就说明对应知识仍有断点。

### 计算机与系统基础

- [ ] 一个在线推理请求从网关到 GPU 再到流式返回，经历了哪些队列、线程和网络交互？
- [ ] 如何区分 CPU、锁竞争、I/O、网络和 GPU 导致的尾延迟？
- [ ] 重试、超时、背压和限流如何配合，哪些设计会导致流量放大？
- [ ] 为什么缓存局部性、内存拷贝和序列化会影响推理服务吞吐？

### 大模型与生成模型原理

- [ ] prefill 和 decode 的计算、显存及并行特征有何不同？
- [ ] KV Cache 的大小由哪些变量决定，GQA/MQA 如何改变它？
- [ ] Transformer 中哪些模块更偏 compute-bound，哪些可能偏 memory-bound？
- [ ] MoE 为什么可能降低计算量却增加系统复杂度？
- [ ] Diffusion 推理与自回归 LLM 推理的主要优化对象有何不同？

### 加速硬件与高性能算子

- [ ] 如何判断一个 Kernel 是 compute-bound 还是 memory-bound？
- [ ] block size、共享内存、寄存器和 occupancy 之间如何权衡？
- [ ] FlashAttention 为什么能降低 attention 的 I/O 成本？
- [ ] 如何验证一个融合算子既正确又比原实现快？
- [ ] 面对不同 shape 和不同硬件，如何避免只优化单个 benchmark？

### 模型级推理优化

- [ ] 权重量化、激活量化和 KV Cache 量化的难点分别是什么？
- [ ] 为什么参数减少或 FLOPs 降低不一定带来端到端加速？
- [ ] PTQ 与 QAT 如何选择，校准集应如何设计？
- [ ] 剪枝、稀疏化、蒸馏和投机解码分别改变了什么成本？
- [ ] 如何建立质量回归与性能收益的联合验收标准？

### 推理引擎与运行时

- [ ] continuous batching 如何提高吞吐，又如何影响延迟？
- [ ] PagedAttention 解决了什么问题，仍有哪些管理开销？
- [ ] KV Cache 如何分配、复用、淘汰，并处理碎片？
- [ ] TP、PP、DP、EP、SP 分别适合什么场景，通信成本在哪里？
- [ ] 如何设计 prefill/decode 分离，什么时候它可能得不偿失？
- [ ] 为推理框架新增模型或硬件后端时，需要哪些接口和测试？

### 分布式推理与生产系统

- [ ] 如何从目标 QPS、输入输出长度和 SLO 估算 GPU 容量？
- [ ] 如何兼顾 GPU 利用率、突发流量和 P99 延迟？
- [ ] Worker 故障、网络分区或新版本性能回退时，系统如何降级和恢复？
- [ ] 多租户共享 GPU 时，如何实现公平性、隔离和成本核算？
- [ ] 哪些指标可以把线上退化定位到调度、引擎、通信或算子？
- [ ] 如何设计压测，使结果能够代表真实流量而非理想 benchmark？
